    Brando: Brand-as-Code

    Brando Schema v1.3 - Governance Framework for AI Brands

    Brando is an open-source markup language and vocabulary for structuring brand identity, governance policies, and content guidelines in machine-readable Markdown/YAML. It enables AI systems and Large Language Models to retrieve and apply brand rules consistently across automated workflows.

    Published under Creative Commons Attribution 4.0 (CC BY 4.0) with W3C Royalty-Free Patent Policy protections, Brando extends Schema.org's Brand vocabulary to support structured brand representation across verbal identity, visual identity, audio branding, strategic positioning, governance, and approved assets.

    Schema.org is the shared structured-data vocabulary founded by Google, Microsoft, Yahoo, and Yandex and developed through an open community process. It is widely used to describe web content for search and discovery.

    Brando builds on that model, but shifts the focus from web and SEO markup to machine-readable brand governance for AI systems.

    Although Brando follows Schema.org and linked-data principles, its primary working format is Markdown with YAML frontmatter. Teams can author, review, and version brand knowledge in Git-friendly files, then emit JSON-LD where linked-data publication, API exchange, or graph tooling needs it.

    What Brando Is

    An open markup language and vocabulary, not a platform, software, or service.

    Brando (brand-oracle) is a markup language for describing brand knowledge in structured Markdown/YAML. It gives teams a shared way to mark up the things a brand governs, the attributes attached to them, the relationships between them, and the rules that control how they are used.

    In practice, Brando markup combines human-readable Markdown with structured YAML frontmatter so brand guidance can be read by people and parsed by machines.

    Brando defines entity types and 90+ brand properties for structuring brand intelligence using five core building blocks:

    • Entities: brand system entities such as brands, products, categories, campaigns, contexts, policies, prompts, personas, and models; and operational entities such as applications, touchpoints, assets, reports, disclosures, claims, and evidence
    • Properties: structured attributes on those entities, including names, definitions, tone, colour, typography, asset URLs, usage guidance, and review workflows
    • Relationships: typed links between entities, such as hasPolicy, hasPrompt, usesVisualToken, appliesToProduct, effectiveDuring, and inheritsTokens
    • Controls: constraints and governance rules that define what is allowed, required, prohibited, reviewed, escalated, inherited, or overridden
    • References: approved examples, discouraged examples, prohibited examples, source documents, design system links, asset libraries, guidelines, evidence, audit records, and benchmark examples

    Together, these terms extend Schema.org's Brand type so brand guidance can be retrieved, interpreted, and applied by AI systems.

    How It Works

    1. Structure Brand Data

    Brand strategists and technical teams use Brando Schema vocabulary to express brand knowledge as:

    • Markdown with YAML frontmatter (primary authoring format, human-readable and Git-friendly)
    • YAML profiles (structured configuration for tooling and CI/CD)
    • JSON-LD (secondary linked-data format for publication, APIs, and graph interoperability)
    • JSON sidecars (for tooling compatibility where needed)

    Example frontmatter:

    ---
brando_type: Brand
name: Acme Corp
toneOfVoice: confident, clear, and approachable
guardRails:
  - Never make medical claims
  - Always cite sources
---

    2. Store as Files

    Brand data can live in version-controlled repositories, with no database or platform required.

    3. Serve to AI Systems

    Organisations implement retrieval layers (RAG systems, MCP servers, API endpoints) to serve structured brand data to AI workflows.

    4. AI Systems Retrieve and Apply

    LLMs, agents, and content generation pipelines query the structured brand data and apply brand rules at runtime.

    No vendor lock-in. The schema is open. The files are portable. Implementation is up to you.

    What Brando Is NOT

    • Not a platform: it's a data vocabulary
    • Not software: it's a schema specification
    • Not a service: it's a standard anyone can implement
    • Not proprietary: it's CC BY 4.0 open source

    Think of it like:

    • Schema.org for product data
    • OpenAPI for API specifications
    • BPMN for process diagrams

    Brando is for machine-readable brand governance.

    Implementation

    Organisations implement Brando-structured brand data in different ways:

    Common patterns:

    • File-based RAG retrieval (Markdown/YAML brand data in Git)
    • MCP servers exposing brand data to AI assistants and agents
    • Custom APIs serving JSON-LD brand data to applications
    • CMS integrations for content governance

    Infrastructure choices:

    • Self-hosted (DigitalOcean, AWS, Azure, on-prem)
    • Managed services (any RAG platform, vector DB, CMS)
    • Hybrid (Git + CDN for static brand data)

    Brando is infrastructure-agnostic, so organisations can use the stack that fits their operating model.
